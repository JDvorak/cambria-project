Cambria, Week Four: This Is Fine

July 6, 2020

I wrote you a nice week three post but didn’t get around to sharing it. You can read it if you like, but briefly, in week three we managed to connect the two ends of the project: we managed to add features to our issue tracker using Cambria.

Once we’d managed to do that, we stepped back, squinted at what we’d done and asked ourselves whether it was the right shape. My dear reader, it was not!

The week three prototype was wrong in three important respects:

  • it didn’t actually solve the right problem
  • it produced incorrect results
  • it couldn’t do a lot of the things we needed

We decided that bugs and missing features were big problems (more on that later), but that if we didn’t build software which addressed an actual human problem found in the field we were just wasting our time.

An Actual Human Problem

Our first few prototypes were designed to demonstrate the concept behind Cambria: that you could have multiple versions of a piece of software running in the field and yet still interoperate despite evolving application types.

If you watch the recorded demos, I think Geoffrey demonstrates this pretty well, with two copies of the same program running side-by-side even in the same window.

Unfortunately, it’s very rare for anyone to run two copies of the same program simultaneously in the same window. It’s much more common for developers to run a single copy at a time and then to have to merge in patches sent by their collaborators, and that didn’t really work at all.

We’d optimized for the experience of our demos (which we took to calling the Researcher Experience, or RX) instead of the developer experience (DX) of building real software.

We made a few changes to fix this, but the biggest was that we now treat our lenses more like a migrations directory, and produce the types your program will use by summing up all the lenses found there. This means that if you pull a branch in from a collaborator you shouldn’t have to take any manual fix-up steps to get their code working.

As we thought through the consequences of this change we realized that our current propagating-lens-patches model was not going to work. It would incorrectly orphan some changes, and worse, the same problem meant that async collaboration was broken too.

Fixing Cambria

The current version of Cambria is becoming increasingly ergonomic. At this point, we think we’re close to solving a useful problem and having a good interface to do so. Therefore, we’ll be returning our attention to the functioning of the system along two key axes:

  1. nesting documents is not implemented
  2. we probably ought to stop losing data

You might (reasonably) think that #2 there is important enough to ignore #1. In fact, because of the division of Chitin (storage format, distribution, types) and Cloudina (lenses and patch translation) we can tackle them in parallel.

Our plan for fixing nesting documents looks a lot like work we’ve seen coming out of Benjamin Pierce’s group at UPenn, with symmetric edit lenses looking promising. We have a few advantages of their usual approach, in particular that because we instrument users’ code directly we don’t have to guess at intent – we can instrument it.

As to the correctness problem, well, we’re trying something new based around a log of all operations. It seems promising thus far, but we’re coming dangerously close to building a new CRDT on top of an existing one and so we need to reconcile that.

Out & About

We had a lovely audience for our last project demo from several folks who responded to an earlier update with interest in our work from their own past histories, including (alphabetically by author) Douglas Creeger’s PhD, Blazon (Brantley Harris), and IPLD Schemas (Eric Myhre).

Q: How do you track “time to live” for a version so you know when you can retire the old one?

In a centralised system, you want to know when enough of your old clients are upgraded to the new version so you can remove support for older versions. The Cambria library works more like an ActiveRecord migrations directory – the code automatically migrates old data up to the new version when it encounters it, so you don’t need to think about deprecating old versions. It can also migrate the data down to older versions to ensure backwards support.

Q: What happens when you’re simply missing data that a new or old version needs?

We can’t invent data for you, but we could detect, for example, if an older version of the software required a certain field and it has become optional in the new version. Cambria, by its nature, forces you to notice and think about this problem as you write your lens migration. You might choose to fill in a default value, or to deliberately crash the old program. You wouldn’t want to send a too-wide license plate to an old stamping machine, for example, or to map missing data to an arbitrary location.
